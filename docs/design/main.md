Android 向け Apple Map 連携基盤 仕様書

1. 目的

Android アプリ上で Apple の地図を扱うため、WebView 上の MapKit JS を用いた地図表示基盤を提供する。
呼び出し側は MapKit JS や WebView の詳細を意識せず、Kotlin の状態モデルとイベントで地図を扱えることを目的とする。

本仕様は特に以下を重視する。
•	Swift / MapKit に近い概念で扱えること
•	Presenter / StateFlow / Compose と相性がよいこと
•	ユーザー操作とシステム操作の競合で破綻しないこと
•	Annotation / Overlay を宣言的に渡せること
•	デフォルト描画とカスタム描画の両方に対応できること

⸻

2. スコープ

本仕様の対象は以下とする。
•	地図表示
•	表示領域の制御
•	Annotation 表示
•	Overlay 表示
•	地図イベントの通知
•	Kotlin 側状態と MapKit JS 側状態の同期ルール

本仕様の対象外は以下とする。
•	経路探索ロジックそのもの
•	地図検索 API の詳細
•	Apple token 発行基盤の詳細実装
•	カスタム描画の具体的な UI 実装
•	クラスタリングやヒートマップなどの高度機能の詳細

⸻

3. 全体構成

構成は以下の三層とする。

3.1 アプリ層

Compose / Presenter / UseCase などのアプリケーションコード。
地図を 状態モデル と イベント で扱う。

3.2 Android Bridge 層

Kotlin と WebView 間の橋渡しを行う。
役割は以下。
•	Kotlin 状態を JS にコマンドとして伝える
•	JS からのイベントを Kotlin イベントに変換する
•	差分更新や反映ループ抑止を担う

3.3 Map 実装層

WebView 上で動作する MapKit JS。
地図描画とユーザー操作の実体を持つ。

⸻

4. 設計方針

4.1 Kotlin 側を主の状態管理とする

地図の表示意図は Kotlin 側が保持する。
MapKit JS 側は描画先かつ操作イベントの発生源として扱う。

4.2 完全な双方向即時同期は行わない

地図操作中に Kotlin と JS を相互に押し返し合うと、ループやガタつきが発生しやすい。
そのため、システム操作は適用対象、ユーザー操作は観測対象 として扱う。

4.3 Swift MapKit に寄せた領域モデルを採用する

初期の表示領域モデルは center + delta を基本とする。
これは Swift MapKit の CoordinateRegion に近い発想であり、iOS 実装との概念整合性を優先する。

4.4 実装詳細は Adapter に閉じ込める

Presenter や呼び出し側は MapKit JS の API 名や WebView 実装を知らなくてよい。
JS 固有概念は Android Bridge 層と Map 実装層に閉じ込める。

⸻

5. 表示領域モデル

5.1 基本方針

表示領域は Kotlin 側で CoordinateRegion 相当のモデルとして扱う。
少なくとも以下の意味を持つ。
•	center: 地図中心座標
•	delta: 緯度・経度方向の表示幅
•	lastOperation: 最終更新主体
•	revision: システム更新の識別子
•	isSettled: 操作途中か確定後か

5.2 lastOperation

表示領域の変更元を区別するため、以下を持つ。
•	interactive
•	ユーザーのパン・ズームによる変更
•	system
•	ボタン操作、初期表示、選択連動などアプリ都合の変更

5.3 同期ルール
•	system 更新は map に適用対象とする
•	interactive 更新は Kotlin 側で受け取るが、即座に map へ再適用しない
•	地図操作中は中間状態を観測できる
•	操作終了時に確定状態として取り扱う
•	自身が発行した更新の反射を区別するため revision を用いる
•	座標や delta の比較は完全一致ではなく誤差許容で扱う

⸻

6. 状態同期の考え方

6.1 単一状態ではなく役割を分ける

アプリ側では以下を論理的に区別する。
•	desired state
•	アプリが表示したい状態
•	actual state
•	map が現在表示している状態

6.2 反映の原則
•	desired が system 由来であり、actual と十分異なる場合のみ map に反映する
•	actual が interactive 由来の場合は、Presenter が観測・採用する
•	ジェスチャ中は map を押し戻さない

6.3 反映ループ回避

以下のための仕組みを前提とする。
•	revision による自己反映識別
•	誤差吸収による微小差分の無視
•	interactive 中の再適用禁止

⸻

7. コンテンツモデル

地図上の表示要素は以下の2系統に分ける。

7.1 Annotation

点ベースの表示要素。
例:
•	ピン
•	マーカー
•	選択地点
•	現在地に準ずる地点表示

7.2 Overlay

線・面ベースの表示要素。
例:
•	経路線
•	エリア
•	円形範囲
•	ポリゴン領域

7.3 共通方針
•	すべて識別子を持つ
•	Kotlin からはデータモデルとして渡す
•	差分更新できる前提で設計する
•	実描画方式は Adapter / Renderer に委譲する

⸻

8. Annotation 仕様方針

8.1 目的

点情報を MapKit っぽい概念で宣言的に扱えるようにする。

8.2 要件
•	座標を持つ
•	選択状態を持てる
•	表示・非表示を制御できる
•	タイトルや補助情報を持てる
•	デフォルト描画が使える
•	必要時にカスタム描画へ差し替えられる

8.3 表示種別

Annotation は少なくとも次の2系統を扱えること。
•	デフォルト表示
•	標準マーカー、標準ピン、標準 callout 等
•	カスタム表示
•	画像差し替えや独自ビュー表現

8.4 役割分担
•	Presenter は Annotation の意味と状態だけを渡す
•	Renderer はその style に応じて実描画を決める

⸻

9. Overlay 仕様方針

9.1 目的

線や面の表現を地図ロジックから独立して宣言的に扱えるようにする。

9.2 要件
•	線・面・円など複数形状を扱える
•	stroke / fill のような基本スタイルを持てる
•	表示・非表示を制御できる
•	z順を制御できる
•	デフォルト描画とカスタム描画を両立できる

9.3 最低限の対象

初期段階では以下を対象とする。
•	polyline
•	polygon
•	circle

⸻

10. デフォルト表示とカスタム表示

10.1 基本方針

表示モデルは 意味情報 と 描画種別 を分離する。
呼び出し側は「何を描きたいか」を渡し、詳細な見た目の責務は描画層に寄せる。

10.2 デフォルト表示

共通実装として提供する。
利用側は最小限の指定だけで表示できる。

10.3 カスタム表示

キーまたは種別を通じて Renderer を選択できる構造とする。
呼び出し側はカスタムビューそのものではなく、style 種別と必要パラメータ を渡す。

10.4 目的

この方式により以下を両立する。
•	呼び出し側 API の安定性
•	画面ごとの見た目差し替え
•	実装詳細の局所化

⸻

11. イベント仕様

地図から Kotlin 側へは、少なくとも以下のイベントを通知できるものとする。

11.1 領域イベント
•	領域変更中
•	領域変更確定
•	操作開始 / 操作終了

11.2 Annotation イベント
•	タップ
•	選択
•	選択解除
•	必要であればドラッグ関連

11.3 Overlay イベント
•	タップ

11.4 イベントの原則

イベントは UI 実装詳細ではなく、意味のあるドメイン寄りイベントとして Kotlin 側へ渡す。

⸻

12. 差分更新方針

12.1 基本方針

Annotation / Overlay は id 単位で管理し、差分更新を前提とする。

12.2 目的
•	毎回全消し再描画しない
•	選択状態変更や一部更新を軽量に扱う
•	WebView ブリッジの更新量を最小化する

12.3 更新単位
•	add
•	update
•	remove

⸻

13. Presenter との関係

13.1 Presenter の責務
•	地図に表示したい状態を作る
•	地図イベントを受けてアプリ状態を更新する
•	画面の他要素と地図状態を同期する

13.2 Presenter が知らなくてよいもの
•	WebView 実装
•	MapKit JS の API 詳細
•	DOM / JavaScript の描画ロジック

13.3 Presenter が扱うもの
•	region
•	annotations
•	overlays
•	selected state
•	地図由来イベント

⸻

14. Compose との関係

Compose 側は地図コンポーネントを 1 つの UI ノードとして扱う。
ただし内部実体は WebView であるため、Compose の再構成と地図の実状態を完全一致で扱うのではなく、Bridge 層による差分適用を前提とする。

重要な点は以下。
•	再構成のたびに地図を再生成しない
•	state 更新のたびに全面再描画しない
•	gesture 中の押し戻しを避ける

⸻

15. 非機能要件

15.1 安定性
•	ユーザー操作中に表示領域が暴れないこと
•	自己反映ループが起きないこと

15.2 拡張性
•	Annotation / Overlay 種別を後から追加できること
•	カスタム描画の差し替えが可能であること

15.3 保守性
•	Swift / iOS 側の概念と大きく乖離しないこと
•	JS 固有事情をアプリ層へ漏らさないこと

15.4 性能
•	不要な全再描画を避ける
•	差分更新を基本とする
•	領域変更イベントの高頻度連打に耐えられること

⸻

16. 初期実装で満たすべき範囲

初期フェーズでは以下を満たせばよい。
•	CoordinateRegion ベースの表示領域制御
•	interactive / system 区別
•	revision による自己反映抑止
•	Annotation のデフォルト表示
•	Annotation のカスタム表示フック
•	Polyline / Polygon / Circle の Overlay 表示
•	タップ系イベント通知
•	差分更新

将来的には以下を拡張対象とする。
•	クラスタリング

⸻

18. まとめ

本基盤は、Android 上で Apple の地図を扱うために、WebView + MapKit JS を実装詳細として隠蔽し、アプリ側には MapKit 風の Kotlin 状態モデル を提供するものである。

中心となる考え方は以下。
•	表示領域は CoordinateRegion ベースで扱う
•	ユーザー操作とシステム操作を区別する
•	双方向完全同期ではなく、破綻しにくい条件付き同期を採用する
•	Annotation / Overlay は宣言的なデータモデルで渡す
•	デフォルト表示とカスタム表示を両立する
•	Presenter は意味モデルだけを扱い、描画実装詳細を知らない
